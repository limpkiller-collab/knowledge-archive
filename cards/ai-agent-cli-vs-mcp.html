배경: AI 에이전트 구현의 딜레마

AI 모델이 외부 시스템과 상호작용하는 방식은 크게 두 가지로 나뉜다. 하나는 수십 년의 역사를 가진 기존 방식인 CLI(Command Line Interface)이고, 다른 하나는 표준화를 추구하는 신규 방식인 MCP(Model Context Protocol)이다.

  • CLI: 터미널 방식의 입출력 인터페이스 — LLM이 직접 셸 명령어 실행
  • MCP: 표준화된 API 인터페이스 — N개 AI 모델과 M개 시스템을 N+M 구조로 연결

각 접근 방식은 서로 다른 장단점을 가지고 있으며, 최적의 선택은 요구사항에 따라 달라진다.

MCP (Model Context Protocol)

정의: N개의 AI 모델과 M개의 외부 시스템 통합 문제를 범용 인터페이스로 해결하는 표준 프로토콜.

비유: "범용 USB-C 포트" — 한 번 표준을 정하면, 어떤 기기든 호환 가능.

✓ 강점

  • 프로세스 격리: 강력한 보안 — 각 시스템이 독립적인 프로세스에서 실행
  • 생태계 표준화: 장기적으로 벤더 중립성 확보
  • 인간 승인 게이트: 중요 작업에 대한 내장된 검증 메커니즘
  • 확장성: 새로운 도구 추가 시 표준 인터페이스만 따르면 됨

✗ 약점

  • Context Bloat: 모든 도구의 스키마를 사전에 로딩 → 15,500+ 토큰 소비
  • Wrapper Tax: 각 시스템별로 전용 MCP 서버 구현 필요
  • 컨텍스트 감소: 실제 추론에 쓸 수 있는 컨텍스트 윈도우 축소
  • LLM 친화성 낮음: 훈련 데이터에 포함되지 않은 새로운 패러다임

CLI with Skills 아키텍처

정의: LLM이 훈련 데이터에서 이미 습득한 유닉스 철학과 터미널 방식을 직접 활용하는 아키텍처.

✓ 강점

  • 네이티브 호환: 번역 레이어 없이 LLM에 직관적
  • 점진적 공개(Progressive Disclosure): 필요할 때만 문서 로딩 (~300 토큰 vs 15,500)
  • 제로 통합 오버헤드: 래퍼 서버 구현 불필요
  • 극적인 비용 절감: 컨텍스트 효율 50배 이상 향상
  • 유연성: 새로운 기능 추가 시 유닉스 철학만 따르면 됨

✗ 약점

  • 보안 취약점: 호스트 권한으로 직접 실행 → 악의적 명령어 실행 위험
  • 샌드박스 필요: E2B, Modal microVM 등 격리 인프라 구축 필수
  • 파괴 위험: rm, rm -rf 등 회복 불가능한 명령어 실행 가능
  • 권한 관리 복잡: 프로세스 레벨 격리가 기본 제공되지 않음

미래 방향 — 하이브리드 아키텍처

MCP와 CLI의 장점을 결합한 하이브리드 접근이 미래의 표준이 될 것으로 예상된다:

  • MCP의 역할: 안전한 데이터 접근 — "요리 재료" 역할
  • CLI의 역할: 효율적인 실행 — "레시피 카드" 역할
  • AAIF 표준화: 벤더 중립성 확보로 생태계 통합
  • 샌드박스 기본화: 격리가 기본값으로 자리잡는 추세
  • 컨텍스트 최적화: 토큰 효율이 핵심 경쟁 요소로 부상

실무 시사점

  • 단순 데이터 조회·읽기: MCP 우선 선택 — 보안과 표준화 확보
  • 복잡한 작업 자동화·실행: CLI 또는 하이브리드 — 효율성 극대화
  • 중요도 높은 작업: MCP의 인간 승인 게이트 활용
  • 규모 있는 시스템: 하이브리드 접근 — 보안과 효율의 균형
  • 보안 요구사항: 샌드박스 격리 필수 고려 (E2B, Modal, Docker)
  • 비용 최적화: 점진적 공개 방식으로 컨텍스트 토큰 절감
